# Python 项目开发标准

> **本文件是项目开发的硬性约束**。违反任意一条，CI 或 pre-commit 拒绝合并。
> 软性建议（命名风格、注释习惯）由 LLM 根据上下文判断，本文件不规定。

---

## 0. 项目骨架（立项时建）

### 0.1 必建目录

| 目录                    | 用途                           |
| --------------------- | ---------------------------- |
| `src/<package_name>/` | 业务代码（**必须 src 布局**，禁止直接放仓库根） |
| `tests/`              | 单元 / 集成测试（与源码平级，不在 `src/` 内） |
| `docs/`               | 项目文档（架构、API、操作指南）            |
| `scripts/`            | 维护脚本（**不入 wheel**）           |

### 0.2 必建文件

| 文件               | 用途                                           |
| ---------------- | -------------------------------------------- |
| `README.md`      | 入门、命令、依赖                                     |
| `pyproject.toml` | PEP 621 元数据 + 构建声明                           |
| `.gitignore`     | Python 标准忽略                                  |
| `.env.example`   | 环境变量样例（**禁止**含真实密钥）                          |
| `CLAUDE.md`      | AI 协作入口，**第一行必须** `@python_project.md` 引用本文件 |

### 0.3 必装 pre-commit hook

所有项目必装，**不依赖平台**。commit 时硬性阻断：

1. `ruff check --fix`
2. `ruff format`
3. CI 第 1-3 项的快速版本

### 0.4 必配 CI（按项目类型选）

CI 配置**按平台**选，缺一不可：

- **GitHub 仓库**：`.github/workflows/ci.yml`
- **GitLab 仓库**：`.gitlab-ci.yml`
- **本地项目**：可不配（仅 pre-commit 兜底）

CI 必跑项：

1. `ruff check .`
2. `ruff format --check .`
3. `pytest tests/ -q`
4. `mypy src/`（项目声明 strict 时必跑）

## 1. 编码规约（违反任一条 = 拒合并）

### 1.1 运行环境

- **必须**用项目虚拟环境（`.venv/`）。
- **禁止**用系统 Python 或全局 site-packages。

### 1.2 代码规模

- **必须**单文件 ≤ 800 行；超须拆分。
- **必须**单函数 ≤ 80 行；超须拆分。
- **必须**单行 ≤ 150 字符。

### 1.3 命名

- 模块 / 函数 / 变量：snake_case。
- 类：PascalCase。常量：SCREAMING_CASE。
- 私有：单下划线前缀。
- **禁止** data / temp / process / handle 等模糊命名（除非范围 ≤ 10 行且注释说明）。

### 1.4 类型注解

- **必须**标注所有生产代码函数的参数和返回类型。
- **必须**显式标注 `-> None`。
- **禁止** lambda 承载复杂逻辑（> 1 个表达式）。

### 1.5 职责与设计

- **必须**一个函数 / 模块 / 类只承担一个主职责。
- **必须**顶层入口函数只编排流程，不承载实现细节。
- **禁止**入口、核心业务、数据模型、工具混在同一模块。

### 1.6 错误处理

- **禁止** bare except（`except:`）。
- **禁止** `except SomeError: pass`（静默吞掉）。
- **必须**为边界操作（IO / 网络 / DB / 子进程 / 第三方）显式捕获具体异常。
- **必须**当前层无法处理时，补充上下文后抛出。
- **禁止**用异常控制正常业务流程。
- **禁止**用 `print` 代替日志。
- **必须**错误日志含异常信息和关键业务上下文。
- **禁止**输出密钥、令牌、密码、手机号、证件号。

### 1.7 文档与注释

- **必须**为公共模块 / 类 / 函数写中文 docstring。
- **必须**为非直观逻辑、业务规则、边界条件写注释，解释"为什么"。
- **禁止**保留大段注释掉的旧代码。

### 1.8 导入

- **必须**导入顺序：标准库 → 第三方 → 本地。
- **必须**不同分组之间空一行。
- **禁止**未使用导入。
- **禁止** `from xxx import *`（项目约定除外）。

### 1.9 数据与状态

- **必须**不可变设计优先。
- **禁止**列表、字典、集合等可变对象作默认参数。
- **必须**跨模块传递的数据结构用 `dataclass` / `TypedDict` / `Pydantic`。
- **禁止**全局可变状态（常量除外）。

### 1.10 测试

- **必须**测试放 `tests/`，命名 `test_*.py`。
- **必须**新功能 / Bug 修复同步补测试。
- **必须**覆盖三类路径：正常 / 边界 / 异常。
- **禁止**提交无法运行或仅供调试的测试。

### 1.11 修改纪律

- **禁止**调试残留、死代码、废弃脚本、临时实现。
- **禁止**重构改变既有行为而不更新测试和文档。
- **禁止**无必要的大重构。
- **必须**修改前先阅读相关文件，遵循现有架构。
- **必须**违反本规范时，commit message 写明 WHY。

## 2. 开发流程（按阶段执行）

### 2.1 开工前

- **必须**澄清需求、出 SPEC。
- **必须**写可执行 plan（含文件级任务分解）。

### 2.2 开发中

- **必须**按 plan 推进，每完成一项打勾。
- **必须**每个新功能先写测试（红 → 绿 → 重构）。

### 2.3 提交前

- **必须** ruff + pytest 全过才能 commit。
- **必须**commit message 遵循 Conventional Commits：`type(scope): 主题`（中文主题，1 行 ≤ 50 字符）。
- **必须**每个 commit 单一意图。

### 2.4 合并前

- **必须**PR 跑过 CI 全绿。
- **必须**自检：lint / test / 文档 / SPEC 同步。
- **必须**code review 通过。

### 2.5 验收后

- **必须**更新 `CHANGELOG.md`（破坏性变更必写迁移指南）。
- **必须**更新 `README.md` / `SPEC.md`（接口变更时）。

---

## 3. 质量验收（交付前必过）

### 3.1 编码

- ruff check 0 error。
- ruff format 0 diff。
- mypy 0 error（项目启用 strict 时）。

### 3.2 测试

- pytest 全过。
- 核心模块（`src/<pkg>/api/` `core/` `services/`）覆盖率 ≥ 80%。
- 工具脚本（`scripts/`）覆盖率 ≥ 60%。

### 3.3 文档

- `README.md` 命令可执行。
- `SPEC.md` 与实现一致。
- `CHANGELOG.md` 含本次版本。

### 3.4 追溯

- 所有违规 commit 在 message 写明 WHY。
- 所有架构决策有 SPEC 文档支撑。
- 所有破坏性变更在 `CHANGELOG.md` 标注。
