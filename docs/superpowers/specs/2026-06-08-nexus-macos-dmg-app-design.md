# Nexus macOS DMG 应用设计规格

> 日期：2026-06-08  
> 范围：macOS 桌面安装包、内置运行时、UI 改版、微信通道入口  
> 原型：[docs/prototypes/nexus-handdrawn-assistant-ui-v2.html](../../prototypes/nexus-handdrawn-assistant-ui-v2.html)

## 目标

Nexus 从“AI 网关”调整为面向个人用户的 macOS 个人 AI 助手。用户在 GitHub Release 下载一个 DMG，双击安装后即可打开 Nexus，不需要理解 Python、虚拟环境、Node、后端端口或命令行。

第一版只支持 macOS，交付物为一个可拖入 Applications 的 `.dmg` 安装包。应用内置前端、后端和 Python 运行时，启动后自动准备本地服务，并通过桌面窗口展示个人任务助手界面。微信能力作为“微信通道”入口，而不是隐藏在插件管理中。

## 用户体验

### 安装体验

用户从 GitHub Release 下载 `Nexus-<version>-mac.dmg`，打开 DMG 后将 Nexus 拖入 Applications。首次启动时，应用自动准备本地后端和数据目录。用户不需要执行安装脚本，不需要安装 Python，不需要运行 `npm install`。

### 首次启动

首次启动只出现必要配置：

- 模型名称，默认 `MiniMax-M3`
- API 地址，默认 `https://api.minimaxi.com/v1`
- API 密钥
- 温度参数，默认 `0.7`
- 测试连接
- 开始使用

高级配置、数据目录、诊断信息放入设置页，不在首启页暴露。

### 主界面

主界面以聊天任务为中心，采用固定左栏 + 主工作区结构：

- 左侧：Nexus 标识、新任务、最近任务、微信通道状态入口、设置入口
- 主区：当前任务标题、用户可理解的任务状态、消息/结果反馈、底部输入框

不展示三栏状态面板，不把底层运行状态、会话压缩、记忆机制、路由和插件逻辑暴露给普通用户。微信来源的任务进入统一“最近任务”，不单独展示微信会话列表。

### 微信通道

微信入口命名为“微信通道”，不使用“插件管理”作为主要文案。微信通道页包含：

- 微信作为随身 IM 通道的说明
- 扫码绑定二维码
- 等待扫码、已绑定、二维码过期、通道运行中、需要重新绑定等状态
- 刷新二维码

微信发起的任务进入统一最近任务列表，并以来源标记提示来自微信。

### 设置页

设置页只展示普通用户能理解的项目：

- 当前模型
- API 地址
- API 密钥
- 测试连接
- 数据与隐私
- 高级设置入口

微信通道不在设置页重复展示。端口、进程、虚拟环境、命令行参数默认不展示。诊断页或高级模式可在后续版本补充。

## 视觉方向

视觉方向为“暖色艺术感个人助手”：温润、克制、安静，有明确品牌记忆点，但不做后台管理感和开发者工作台气质。

设计原则：

- 使用暖色纸感背景、柔和深绿侧栏和细腻线条，形成舒适的个人助理气质。
- 微信绿色仅用于微信状态与通道入口，不铺满整体界面。
- 避免后台管理感，减少技术术语。
- 信息密度适中，聊天区留白充足，输入框像“任务委托区”。
- 首屏要表达“个人 AI 助手”，而不是“服务控制台”。

原型文件 `docs/prototypes/nexus-handdrawn-assistant-ui-v2.html` 是本阶段唯一视觉参考，后续实现时不要求逐像素复刻，但要保持信息架构、命名和整体气质一致。

## 技术架构

### 桌面壳

使用 Electron + TypeScript 实现 macOS 桌面壳。

职责：

- 创建主窗口
- 管理本地后端进程生命周期
- 等待后端健康检查通过后加载 UI
- 在应用退出时停止内置后端
- 处理数据目录、日志目录、应用资源路径
- 为后续自动更新、菜单、托盘能力预留边界

Electron 主进程不得承载业务逻辑。业务逻辑仍由 FastAPI 后端和 React 前端负责。

### 前端

继续使用现有 React + TypeScript + Vite。前端构建产物由 Electron 加载。桌面版 UI 改造应优先复用现有接口：

- `/api/sessions`
- `/api/model`
- `/api/models`
- `/api/channels/wechat/*`
- `/api/ws`

前端需要新增或调整：

- 首启配置视图
- 个人任务助手主界面
- 微信通道独立页面
- 设置页
- 后端启动中、启动失败、健康检查失败状态

### 后端

继续使用现有 FastAPI 后端。桌面版不改变核心 API，只增加必要的桌面运行适配。

后端运行要求：

- 绑定 `127.0.0.1`，不默认监听 `0.0.0.0`
- 端口优先使用固定默认端口，冲突时由桌面壳选择备用端口并传给前端
- 使用应用管理的数据目录，默认位于 macOS 用户应用数据目录或兼容的 `~/.nexus`
- 保持现有 SQLite、会话、模型配置、微信通道逻辑

### 内置运行时

DMG 内置后端运行能力。推荐后端打包为单独可执行文件，放在 `.app/Contents/Resources/` 下，由 Electron 主进程启动。

构建策略：

- 开发期使用项目 `.venv` 和源码启动，方便调试。
- 发布期把 Python 后端、依赖和必要资源打包为独立后端可执行文件。
- Electron 应用只通过进程和 HTTP/WebSocket 与后端通信。

这样可以避免让普通用户安装 Python 或创建虚拟环境。

## 数据与配置

桌面版保留本地优先原则。模型配置、会话、微信绑定信息和日志均保存在用户本机。

配置原则：

- API 密钥不出现在日志中。
- UI 中 API 密钥默认使用密码输入框。
- 诊断信息不得输出密钥、令牌、手机号、证件号等敏感信息。
- 后端读取配置时继续兼容环境变量和现有配置文件。

第一版可以继续兼容 `~/.nexus`，但 Electron 主进程需要显式设置 `NEXUS_HOME`，避免不同启动方式的数据目录不一致。

## 错误处理

桌面版需要覆盖以下失败状态：

- 后端启动失败
- 后端健康检查超时
- 端口被占用
- 模型连接测试失败
- API 密钥缺失
- 微信二维码获取失败
- 微信二维码过期
- 微信通道断开

错误提示要面向用户说明“现在该做什么”，而不是只展示异常堆栈。详细日志放入诊断入口。

## GitHub Release 交付

GitHub Actions 只服务于开发发布，不面向最终用户。发布流程：

1. 在 GitHub 创建版本标签或 Release。
2. macOS runner 构建前端。
3. 打包后端可执行文件。
4. 打包 Electron 应用。
5. 生成 `.dmg`。
6. 上传 `.dmg` 到 GitHub Release。

最终用户只看到一个 DMG 下载文件。

## 测试与验收

### 单元与集成测试

- 后端改动使用项目 `.venv` 运行 pytest。
- 前端改动运行 TypeScript 构建和必要的组件/端到端检查。
- 桌面壳需要覆盖后端进程启动参数、健康检查、退出清理的测试或脚本验证。

### 手动验收

第一版 DMG 验收标准：

- 干净 macOS 环境中安装后可打开。
- 用户不需要安装 Python、Node 或执行命令行。
- 首次启动可以配置模型并测试连接。
- 用户可以在主界面发起任务并接收状态与结果反馈。
- 微信通道可以进入扫码绑定流程。
- 关闭应用后内置后端进程被清理。
- 重新打开应用后会话、配置和微信状态可恢复。
- GitHub Release 中能直接下载 `.dmg`。

## 非目标

第一版不做：

- Windows/Linux 安装包
- 自动更新
- 代码签名与 notarization 的完整商业发布流程
- 多用户账户体系
- 复杂插件市场
- MCP 可视化管理后台
- 原生 Swift 重写
- 完整离线 LLM

## 实施顺序建议

1. 建立 Electron + TypeScript 桌面壳。
2. 实现开发模式后端启动与健康检查。
3. 接入现有前端构建产物。
4. 改造 UI 为首启配置、个人任务助手、微信通道、设置四个视图。
5. 实现发布模式后端打包。
6. 实现 Electron DMG 构建。
7. 添加 GitHub Actions Release 构建流程。
8. 做干净环境安装验收。

## 决策记录

- 只支持 macOS。
- 使用 Electron + TypeScript 作为桌面壳。
- DMG 内置后端运行时。
- 微信能力命名为“微信通道”，作为 IM 通道入口。
- 用户下载和安装只接触一个 DMG 文件。
